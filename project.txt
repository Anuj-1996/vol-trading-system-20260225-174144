# Vol Trading System - Architecture & Performance Analysis

## 1. Backend Architecture
The backend is a hybrid C++/Python system designed for high-performance financial analytics.

### Components in C++ (`backend/cpp/vol_core.cpp`)
- **Heston Monte Carlo**: Multi-threaded path simulation (ARM NEON optimized).
- **FFT Pricing**: Heston characteristic function and fast pricing of strikes.
- **Greeks**: Vectorized Black-Scholes Delta, Gamma, Vega, Theta.
- **Strategy Evaluation**: Payoff calculation, PnL statistics, VaR, and Expected Shortfall.
- **Dynamic Hedging**: Euler-step based hedging simulations.
*Communication*: Native Python bindings via `pybind11`.

### Components in Python
- **API Layer**: FastAPI handles routing and request validation.
- **Services**: Orchestration (`engine_service.py`), async job management (`job_service.py`), and analytics (`positioning_service.py`).
- **Data Pipeline**: NSE fetching, cleaning, and storage in SQLite (`vol_engine.db`).
- **AI Agents**: Multi-agent system (Gemma-based) for trade analysis and briefing.

---

## 2. Request Flow: "Fetch and Analyse"
1. **Trigger**: User clicks button in UI.
2. **Fetch**: Frontend calls `/data/fetch-live`. Node/Python fetches raw JSON from NSE.
3. **Caching**: Python cleans data and stores it in an in-memory `_live_data_cache` (keyed by `data_id`).
4. **Analysis Pipeline**:
   - **Positioning**: Runs immediately (Sync) to calculate GEX/VEX/CEX and Gamma Walls.
   - **Static Pipeline**: Calibrates Heston surface (Scipy/C++), ranks strategies (Python), and computes Greeks (C++).
5. **Parallelism**: Calibration uses L-BFGS-B (Scipy), while Monte Carlo and Strategy Evaluation use C++ `std::thread` concurrency.

## 3. API Structure (v1)
- `POST /data/fetch-live`: Fetches NSE data. Synchronous (waits for NSE).
- `POST /positioning/calculate`: Calculates dealer positioning. Synchronous.
- `POST /pipeline/live-static`: Runs full analysis. Synchronous.
- `POST /pipeline/dynamic/submit`: Long-running hedging analysis. **Asynchronous** (returns `job_id`).
- `GET /jobs/{id}`: Polls status of async jobs.
- `POST /ai/chat`: Interactive agent queries. Supports streaming via `/stream`.

## 4. Data Pipeline
- **Source**: NSE India (via `NSEClient` and `NSEOptionChainFetcher`).
- **Processing**: 
  - `NSEDataCleaner` filters illiquid strikes and handles corporate actions.
  - `SurfaceBuilder` generates the volatility grid.
  - `JointHestonCalibrator` fits model parameters to the market surface.
- **Storage**: Real-time snapshots stay in RAM; historical trades and strategy logs go to `vol_engine.db`.

---

## 5. Page Loading Architecture
- **Market Page**: Combines GEX/VEX data with price history from `yfinance`.
- **Vol Surface**: Receives a pre-computed grid of (Strike, Maturity, IV) from the Heston calibrator.
- **Risk Lab/Screener**: Backend computes all strategies in a **single burst job** during the static pipeline, then frontend filters/ranks.

## 6. Performance Characteristics
- **CPU Usage**: Intensive during Calibration and MC. C++ core utilizes all 8+ cores of the host machine.
- **Memory**: Moderate (~200MB - 500MB). C++ uses zero-copy numpy views where possible.
- **Expected Runtime**: 
  - Fetch + Clean: 1-2s.
  - Calibration: 2-5s.
  - Full Strategy Scan (MC): 3-8s.
  - Total "Fetch & Analyse" cycle: **< 15 seconds**.

---

## 7. Scalability & Bottlenecks
- **Multiple Users**: Supported via FastAPI/Uvicorn.
- **Scale (10-100 Users)**: 
  - **Bottleneck 1: NSE IP Blocking**. The system currently uses direct fetching; 100 concurrent users will trigger NSE rate limits.
  - **Bottleneck 2: Python GIL**. While C++ is multi-threaded, the Python orchestration layer can block.
  - **Bottleneck 3: CPU Exhaustion**. 100 concurrent Heston calibrations (120 iterations each) will max out the server.
- **Scale (1000 Users)**: Requires a distributed worker architecture (Celery/Redis) and a centralized data proxy for NSE.
