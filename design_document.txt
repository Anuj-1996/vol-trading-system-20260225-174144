PROJECT DOCUMENT
NIFTY STOCHASTIC VOLATILITY STRATEGY OPTIMIZATION SYSTEM
Version 3.0
Scope: Personal Institutional Grade Risk Engine

⸻

	1.	SYSTEM OVERVIEW

⸻

1.1 Objective

Build a backend driven quantitative system with frontend interface that:
	•	Performs joint Heston calibration across term structure
	•	Uses Carr Madaan FFT for fast pricing
	•	Filters illiquid strikes
	•	Handles weekly and monthly expiries
	•	Generates and evaluates all supported option strategies
	•	Supports static and dynamic hedging
	•	Detects model vs market surface misfit
	•	Ranks strategies based on regime aware risk adjusted scoring

Target underlying: NIFTY index options

Primary language: Python
Frontend: React
Backend API: FastAPI
Database: PostgreSQL or SQLite for local

Input:

3.2.4 Joint Heston Calibration Engine

Model:

Heston stochastic volatility model

Pricing:

Carr Madaan FFT

Joint Calibration Objective:

Minimize weighted squared error across:

All strikes
All expiries simultaneously

Decision Variables:

kappa
theta
xi
rho
v0

Constraints:
	•	Feller condition soft constraint
	•	Parameter bounds

Objective Function:

Weighted RMSE between model IV and market IV

Weights:

Liquidity weighted
Short maturity higher weight

Output:

JointHestonParameters
CalibrationErrorMetrics

Store calibration history for stability monitoring.

3.2.5 Model vs Market Diagnostics

Compute:
	•	Residual surface
	•	Wing mispricing
	•	Term structure deviation
	•	Skew slope mismatch

Metrics:

RMSE per expiry
Systematic bias detection

Flag if calibration unstable.

3.2.8 Static Evaluation Engine

For each StrategyObject:

Loop through terminal_prices.

Compute:

Expected Value
VaR 95
VaR 99
Expected Shortfall
Probability of loss
Return on Margin
PnL Skewness
Convexity Exposure

Store in StrategyMetrics table.

⸻

3.2.9 Dynamic Hedging Engine

User selectable:
	•	No hedge
	•	Daily delta hedge
	•	Threshold hedge

Process:

For each path:

At each time step:
	•	Compute model delta
	•	Adjust hedge
	•	Subtract transaction cost
	•	Track cumulative PnL

Output:

DynamicPnLDistribution

Heavy compute. Must allow async processing.

⸻

⸻

3.2.10 Ranking Engine

Score Function:

Score =
Normalized EV
− lambda1 * ES99
− lambda2 * FragilityScore
	•	lambda3 * ReturnOnMargin

Lambda values depend on regime state.

Top N strategies stored.

3.2.11 Backtesting Engine

Walk forward logic:

For each historical date:
	•	Ingest data
	•	Calibrate
	•	Generate strategies
	•	Select top strategy
	•	Compare realized expiry PnL

Metrics:

Sharpe
Max Drawdown
Tail error rate
Calibration drift

Out of sample validation mandatory.
